# 2026-05-12 개발일지

## 오늘의 한 줄

퇴근 후 Codex harness 템플릿에 phase metadata validator를 추가했다. 이제 phase를 실행하기 전에 구조가 깨져 있는지 먼저 확인할 수 있다.

## 왜 이 작업을 했나

어제까지는 `scripts/create_phase.py`로 phase와 step 파일을 자동 생성할 수 있었다. 하지만 생성된 phase가 실제로 실행 가능한 상태인지 검증하는 장치는 없었다.

Codex harness는 작은 step 파일을 순서대로 실행하는 구조다. 그래서 `index.json`에 적힌 step 번호와 실제 `stepN.md` 파일이 어긋나거나, status 값이 잘못 들어가면 실행 중간에 실패한다. 이런 문제는 실행 전에 잡는 편이 낫다.

오늘 작업의 목적은 단순했다.

- phase를 만들 수 있게 되었으니, 이제 phase가 정상인지 검사한다.
- executor에 더 복잡한 기능을 붙이기 전에 metadata 안정성을 확보한다.
- 이후 `--dry-run`, phase report, hook profile 같은 기능이 재사용할 수 있는 검증 기반을 만든다.

## 작업한 내용

새 스크립트 `scripts/validate_phase.py`를 추가했다.

이 명령으로 phase metadata를 검사할 수 있다.

```bash
python3 scripts/validate_phase.py {phase-name}
```

validator가 확인하는 항목은 다음과 같다.

- `phases/index.json`에 해당 phase가 등록되어 있는지
- `phases/{phase}/index.json` 파일이 있는지
- phase 이름이 디렉터리 이름과 일치하는지
- step 번호가 `0`부터 빠짐없이 이어지는지
- 각 `stepN.md` 파일이 실제로 존재하는지
- status 값이 `pending`, `completed`, `error`, `blocked` 중 하나인지
- `completed` step에는 `summary`가 있는지
- `blocked` step에는 `blocked_reason`이 있는지
- `error` step에는 `error_message`가 있는지

테스트도 함께 추가했다.

- 정상 phase metadata는 통과한다.
- registry 누락, phase 이름 불일치, 잘못된 status, 누락된 step 파일을 오류로 잡는다.
- CLI가 invalid phase에 대해 exit code `1`을 반환한다.

## 구현하면서 신경 쓴 점

이번 작업은 “실행을 막는 도구”라서 오류 메시지가 중요했다. 단순히 실패만 알려주면 다음 행동을 결정하기 어렵다. 그래서 validator는 가능한 한 여러 오류를 한 번에 모아서 보여주도록 했다.

또 하나 신경 쓴 부분은 step 파일 검사 방식이다. 처음에는 metadata에 적힌 step 번호 기준으로만 파일을 확인했는데, 그러면 step 번호 자체가 `1, 2, 3...`처럼 밀려 있을 때 `step0.md` 누락을 놓칠 수 있다. 테스트가 이 문제를 잡아냈고, 기대 번호 `0..N` 기준으로 step 파일을 확인하도록 수정했다.

이런 작은 검증 도구일수록 테스트를 먼저 두는 게 효과적이었다. 원하는 실패 형태가 명확해지니 구현 범위도 자연스럽게 좁아졌다.

## 추가 작업: Executor Dry Run

위 다음 작업을 이어서 진행했다. `scripts/execute.py`에 `--dry-run` 옵션을 추가해서 실제 실행 전에 phase 상태를 읽기 전용으로 확인할 수 있게 했다.

dry-run은 다음 정보를 출력한다.

- validator 결과
- 대상 branch 이름
- 다음 pending step
- step prompt 파일
- 이전 completed step summary
- executor가 읽을 파일 목록
- `codex exec` 실행 개요

의도적으로 하지 않는 일도 명확히 했다.

- Codex를 호출하지 않는다.
- branch를 checkout하거나 생성하지 않는다.
- phase index에 timestamp를 쓰지 않는다.
- step output JSON을 만들지 않는다.

테스트는 `scripts/test_execute.py`에 추가했다. 특히 git repository가 아닌 임시 디렉터리에서도 dry-run이 성공하는지 확인해서, 이 경로가 branch checkout에 기대지 않는다는 점을 고정했다.

추가 검증으로 아래 명령이 통과했다.

```bash
python3 -m unittest discover -s scripts -p 'test_*.py'
.githooks/pre-commit
```

다음 우선순위는 `Phase Report`다. dry-run이 실행 전 preview라면, report는 실행 전후의 phase 상태를 JSON을 직접 열지 않고도 읽는 도구가 된다.
