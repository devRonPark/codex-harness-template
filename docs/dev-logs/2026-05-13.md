# 2026-05-13 개발일지

## 오늘의 한 줄

Codex harness 템플릿에 review rubric을 추가하고, 실제 작은 앱 생성 흐름을 돌려 보면서 "어디서 멈출지"를 판단하는 기준과 현재 템플릿의 다음 개선점을 확인했다.

## 왜 이 작업을 했나

처음 문제의식은 기술적인 것보다 운영적인 것이었다.

하네스를 계속 개선하다 보면 기능을 어디까지 넣어야 하는지 애매해진다. "harness engineering이 요즘 중요하다"는 막연한 동기만 있으면, 기능 추가가 계속 이어지고 멈춤 기준이 사라진다.

그래서 오늘은 템플릿에 더 큰 기능을 붙이기 전에 다음 질문을 먼저 정리했다.

- 어떤 변경은 반드시 고쳐야 하는가?
- 어떤 변경은 기록만 하고 넘어가도 되는가?
- 언제 "이번 step은 완료"라고 말할 수 있는가?
- 이 템플릿이 실제 작은 결과물을 만드는 데 쓸 수 있는가?

오늘의 목표는 Codex harness를 더 화려하게 만드는 것이 아니라, 결과물을 믿고 멈출 수 있는 기준을 추가하는 것이었다.

## 작업한 내용: Review Rubric

`.agents/skills/review/SKILL.md`에 personal review rubric을 추가했다.

큰 방향은 Ouroboros 프로젝트의 code review 형식에서 참고했다. 전체 아키텍처나 철학을 따라가지는 않고, 다음 요소만 작게 가져왔다.

- findings-first review
- severity-ranked findings
- verification evidence
- residual risks
- recommendations

템플릿에는 `P0`부터 `P3`까지의 기준으로 정리했다.

```text
P0: 실행 안전성을 깨는 문제
P1: 결과 신뢰도를 깨는 문제
P2: 유지보수 또는 후속 확인이 필요한 문제
P3: 완료를 막지 않는 polish
```

완료 규칙도 명확히 했다.

- `P0` 또는 unresolved `P1`이 있으면 완료로 보지 않는다.
- `P2`는 지금 고치거나 residual risk로 기록한다.
- `P3`는 기본적으로 scope를 늘리는 이유가 되지 않는다.
- validation evidence가 충분하고 `P0`/`P1`이 없으면 멈춘다.

이 기준은 예를 들어 "구글, 카카오 로그인만 구현" 같은 요청에서 범위 이탈을 막기 위한 장치다. 페이스북, 애플 로그인까지 구현하면 scope creep이다. 반대로 구글/카카오 로그인 UI만 만들고 실제 callback이나 검증이 빠져도 완료가 아니다.

관련 문서도 함께 갱신했다.

- `README.md`
- `docs/HARNESS_TEMPLATE_SUMMARY.md`
- `docs/TEMPLATE_EVOLUTION.md`
- `docs/HARNESS_ADVANCEMENT_ROADMAP.md`
- `docs/NEXT_PROGRESS_PLAN.md`
- `docs/ADR.md`

python3 scripts/report_phase.py harness-smoke

이 과정에서 중요한 문제를 발견했다.

## 발견한 문제: Broad Git Staging

기존 `scripts/execute.py`는 step 완료 후 커밋할 때 `git add -A`를 사용했다.

이 방식은 이번 step에서 생성한 파일뿐 아니라, 실행 전부터 작업트리에 있던 unrelated untracked file까지 함께 stage할 수 있다.

실제 당시 작업트리에는 다음 파일이 있었다.

```text
?? .claude/
```

이 상태에서 phase를 실행하면 `.claude/` 같은 local-only 파일이 step commit에 섞일 위험이 있었다. 실행 자체가 깨지는 문제는 아니지만, 결과 커밋을 믿기 어렵게 만들기 때문에 `P1`로 분류했다.

해결은 두 가지로 했다.

1. `.claude/` 디렉터리를 삭제했다.
2. executor에 clean worktree 정책과 changed-path staging을 추가했다.

이제 `scripts/execute.py`는 phase 실행 전에 worktree가 dirty이면 중단한다. 또한 실행 중 관찰된 changed path만 stage하고, repository-wide `git add -A`를 사용하지 않는다.

회귀 테스트도 추가했다. `.claude/plan-progress.md` 같은 미추적 파일이 있으면 executor가 실행을 시작하지 않고 exit code `3`으로 중단하는지 확인한다.

작업은 아래 커밋으로 정리했다.

```text
33d3258 Harden executor worktree staging
```

이 커밋은 `main`에 fast-forward merge했다.

## Persona Smoke에서 얻은 개선 의견

이번 smoke에서 확인한 가장 큰 병목은 executor 자체가 아니라 phase와 step 작성 UX였다.

### 1. Step Template v2가 필요하다

`create_phase.py`가 생성하는 기본 `step0.md`는 아직 `TBD`가 많다. 30분 안에 작은 결과물을 만들고 싶은 사용자는 여기서 바로 막힐 수 있다.

다음 섹션을 기본 skeleton에 넣는 편이 좋다.

```text
Files To Read
Files To Edit
Expected Output
Acceptance Criteria
Evidence
Recovery
Do Not
```

작은 앱에서는 편의성 문제지만, 큰 앱에서는 step 품질을 유지하는 핵심 장치가 될 가능성이 높다.

### 2. Progress 출력이 너무 시끄럽다

executor spinner가 긴 실행 중 터미널에 매우 많이 찍혔다. 사용자는 정상 진행인지, 멈춘 것인지 판단하기 어렵다.

개선 방향:

- 5초 또는 10초 단위 summary log
- `--quiet-progress` 옵션
- 현재 step runtime과 output artifact path를 간결하게 표시

### 3. elapsed time 표시가 부정확하다

실제 Codex 실행은 약 4분 걸렸지만, executor는 step 완료 시 `[0s]`로 표시했다.

이건 `progress_indicator`의 elapsed 값을 context 종료 전에 읽는 구조 때문으로 보인다. 작은 문제처럼 보여도, 긴 phase에서는 실행 시간 판단을 흐리므로 고치는 편이 좋다.

### 4. Clean worktree 정책은 맞지만 안내가 더 필요하다

dirty worktree면 실행을 막는 정책은 맞다. 다만 persona smoke에서는 "준비 문서를 커밋하고 실행해야 한다"는 점이 사용자에게 명확히 안내되지 않았다.

개선 방향:

- dirty paths 출력
- `commit`, `stash`, `remove` 중 다음 행동 안내
- phase 준비 파일만 남아 있을 때는 "plan commit 먼저 만들기"를 제안

### 5. Output artifact 정책이 애매하다

`step0-output.json`은 실제로 생성되고 report에도 artifact로 표시된다. 하지만 `.gitignore` 때문에 커밋되지는 않는다.

이 정책 자체는 괜찮지만, 문서 표현은 더 명확해야 한다.

```text
raw output은 local artifact다.
durable history가 필요하면 phase run ledger를 사용한다.
```

아직 ledger가 없으므로, 큰 앱에서는 나중에 추적성이 부족해질 수 있다.

## 작은 앱과 큰 앱 기준의 차이

오늘 smoke는 작은 단일 HTML 앱이었다. 그래도 나온 개선 의견 대부분은 큰 앱에서도 의미가 있다.

작은 앱에서 특히 중요한 것:

- single-step MVP scaffold
- step 작성 UX
- 빠른 dry-run과 report

큰 앱에서 더 중요해질 것:

- Step Template v2
- Phase Run Ledger
- progress/status reporting
- clean worktree UX
- recovery/resume flow

따라서 다음 작업은 `Step Template v2`가 가장 자연스럽다. 작은 앱과 큰 앱 양쪽 모두에 효용이 있고, 지금 템플릿의 실제 병목과도 맞아떨어진다.

## 오늘의 결론

현재 harness template은 작은 결과물을 실제로 끝까지 만들 수 있는 상태다.

하지만 30분 사용자 기준으로는 "실행기"보다 "좋은 step을 빠르게 쓰는 경험"이 부족하다. 오늘 추가한 review rubric은 멈춤 기준을 만들었고, executor staging hardening은 커밋 신뢰도를 높였다.

다음 개선은 더 큰 orchestration이 아니라 step 작성 품질을 높이는 쪽이어야 한다.

## Step Template v2 이후 다음 작업

다음 우선순위는 `Phase Run Ledger`다.

이제 step prompt 자체에는 실행 범위, 기대 결과, 증거, 복구 맥락을 적을 자리가 생겼다. 다음 병목은 실제 executor가 phase를 돌릴 때 어떤 시도가 있었고, 어떤 상태 전이가 있었고, 얼마나 걸렸는지를 durable하게 남기는 것이다.

다음 브랜치 후보:

```text
feat-phase-run-ledger
```

핵심 범위:

- `phases/{phase-name}/events.jsonl` 추가
- executor가 step attempt start/end, exit code, status transition, duration을 append-only로 기록
- `scripts/report_phase.py`가 최근 attempt와 retry history를 보여주도록 확장
- raw `stepN-output.json`은 local artifact이고, durable history는 ledger가 담당한다는 정책 문서화

## 내일 세션을 위한 Phase Run Ledger 핸드오프

내일은 구현부터 바로 들어가기보다, 먼저 테스트로 ledger contract를 고정하는 것이 좋다.

시작 체크:

내일 먼저 읽을 파일:

- `docs/NEXT_PROGRESS_PLAN.md`
- `docs/HARNESS_ADVANCEMENT_ROADMAP.md`
- `scripts/execute.py`
- `scripts/report_phase.py`
- `scripts/test_execute.py`
- `scripts/test_report_phase.py`
- `.gitignore`

첫 번째 테스트 목표:

- executor가 step attempt 시작과 종료를 `phases/{phase-name}/events.jsonl`에 append한다.
- report가 ledger가 있을 때 최근 attempt를 보여준다.
- ledger가 없을 때 기존 report output은 깨지지 않는다.

권장 event schema:

```json
{
  "timestamp": "2026-05-13T22:18:50+0900",
  "event": "step_attempt_finished",
  "phase": "example-phase",
  "step": 0,
  "step_name": "project-setup",
  "attempt": 1,
  "status_before": "pending",
  "status_after": "completed",
  "exit_code": 0,
  "duration_seconds": 12,
  "message": "Created project scaffold."
}
```

중요한 범위 제한:

- raw output artifact 정책은 유지한다. `phases/**/step*-output.json`은 `.gitignore`에 남긴다.
- `events.jsonl`은 durable history이므로 ignore하지 않는다.
- executor configuration, worktree isolation, parallel execution은 이번 브랜치에서 하지 않는다.
- 외부 dependency를 추가하지 않는다.
